Net result: we write ~40% less code and get instant sync instead of 10-second polling. The previous Supabase removal was the edge function proxy (misuse). Using Supabase as an actual database is what it's built for.

Intent

Three people on the auction floor, same inventory, same day. When Scotty marks a Teryx "interested" with a $4,200 max bid, the other two phones update instantly — no refresh, no waiting. One person runs a deep comp search, everyone gets the results.

Secondary: comp searches are locked to hardcoded regions. Add controls for radius, regions, and results count.

Scope

In Scope

  • Real-time item overlay sync — status, notes, maxBid, buddyTag push instantly across devices
  • Shared comps cache — one Apify search, all devices benefit
  • Room system — 6-digit code, no auth, 24-hour expiry
  • Presence — see who's connected ("Scotty, Joey, +1")
  • Offline resilience — queue edits, flush on reconnect
  • Search settings — radius, regions, results-per-source

Out of Scope

  • User accounts / passwords — room code is enough for a team tool
  • Full item sync (year, make, model, VIN) — stays local, overlay only
  • Chat between users

Architecture

Supabase Tables

TablePurposeKey Columns
roomsTeam sessionid, room_code (unique 6-digit), created_by, created_at, expires_at
item_overlaysShared item stateid, room_code, item_number, status, max_bid, note, buddy_tag, updated_by, updated_at
shared_compsShared comp cacheid, room_code, query_hash, query_text, results_json, fetched_by, fetched_at
activityChange feedid, room_code, item_number, field, old_value, new_value, user_name, created_at

RLS disabled — team tool, not public. The room_code scopes all data. No row-level security complexity needed.

Sync Flow

  • On edit: Save to IndexedDB (local) → upsert to item_overlays (Supabase) → real-time broadcast triggers on other devices
  • On subscribe: supabase.channel('room:ABC123').on('postgres_changes', ...) — instant push when any row in the room changes
  • On connect: Pull all item_overlays for room → merge into local IndexedDB
  • Presence: channel.track({ user: 'Scotty' }) — shows who's in the room
  • Offline: Edits queue in localStorage → on reconnect, flush queue → Supabase handles ordering by updated_at

Phases

Radius slider, region picker, results-per-source in Settings. Wire through to fetch-comps. No backend change. Immediate value.

Phase 2 — Supabase Setup Free Tier

Unpause existing Supabase project. Create 4 tables via SQL editor in dashboard. Install @supabase/supabase-js. Add env vars to Vercel. Verify connection.

Phase 3 — Real-time Sync Core Feature

Room create/join. Subscribe to real-time changes. Hook saveItem to push overlays. Shared comps cache. This is the money phase.

Phase 4 — Sync UX + Offline Polish

Room UI in Settings. Sync status dot in header. Activity feed. Offline queue. Presence (who's connected).

Constraints

  • Supabase free tier: 500MB database, 2GB bandwidth, 50K monthly active users, real-time connections included. Way more than 3 users need.
  • No auth: Room code is the only gate. Fine for team use.
  • Overlay only: Full item data stays local. Sync payload is small — just status, maxBid, note, buddyTag per item.
  • Last-write-wins: Supabase upsert by updated_at. Activity table preserves history.
  • 24-hour rooms: Auto-expire. Each auction day = fresh room.
  • Existing Vercel API stays: fetch-comps, fetch-npa-inventory, scan-tag remain on Vercel. Supabase handles only sync.
